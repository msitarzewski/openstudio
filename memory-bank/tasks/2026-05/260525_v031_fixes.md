## Patterns Applied

- **Multipart parser must handle the last form field with no per-part `Content-Length`** — browser FormData doesn't include Content-Length per part, and the original `handleExportClean` parser only looked for the next regular boundary marker. When the last part was a small text field like `outputFormat=mp3`, contentEnd ran to `body.length` and pulled the trailing `--<boundary>--\r\n` into the content. Result: `formatParam` became `"mp3\r\n--..."`, never matched `"mp3"`, and the MP3 transcode block was silently skipped. Fix: when no per-part Content-Length is present, take `Math.min(nextRegularBoundary, endBoundary)` and trim the trailing CRLF. Applied at `server/server.js:142-159`. New `handleExportZip` uses the same pattern from the start.
- **Optional integrations must be env-driven, not IP-driven** — show-notes generator previously hardcoded a private dev-machine IP and a specific model. Pattern: define `const X = process.env.X || '<sensible default>'` at the top of the module; all call sites reference the constant. Default chosen to match the most common OSS setup (LM Studio's default `:1234`) so anyone running LM Studio locally "just works".
- **Frontend fallback on optional server endpoints** — `bundleAndDownload` POSTs to `/api/export/zip` for the nice single-zip experience but, on any failure, calls the existing `downloadAll()` to preserve the user's ability to retrieve their files. The new behavior is an additive enhancement, not a replacement.

## Architectural Decisions

- **Decision**: Make the LLM endpoint and model OpenAI-compatible env vars rather than special-casing LM Studio
- **Rationale**: The chat completions JSON contract is the same across LM Studio, Ollama (with the OpenAI shim), vLLM, llama.cpp server, etc. One env-var pair (`LLM_BASE_URL` + `LLM_MODEL`) covers all of them. No vendor coupling, no library dependency, no SDK.
- **Trade-off**: Any dev environment whose LLM lives on a non-default host/port now needs an explicit `LLM_BASE_URL` entry in `.env` to override the localhost default. Acceptable — explicit beats hidden.

- **Decision**: Server-side zip bundling instead of client-side via JSZip-style library
- **Rationale**: `archiver` was already a dependency. Mirrors the existing `/api/export/clean` round-trip pattern. Keeps the browser doing browser things (UI + media capture) and the server doing server things (file munging). No new client deps, no bundler dance.
- **Trade-off**: Uploading all tracks consumes server memory proportional to recording length. Cap is 500 MB total per request (mirrors `handleTranscribe`). For longer sessions, per-track download via the existing `downloadAll()` path is the fallback.
